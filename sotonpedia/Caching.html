[edit] Cache

Smaller amount of advanced memory that improves performance of accesses to larger amounts of slower, less expensive memories

  • Properties
    • Block size: The block size of cache is the same as the block size for main memory
  • Set: Set of memory locations that can store something for fast access
  • Stores
    • Data
    • Instructions
  • Increases performance
    • Frequently running code: Frequently running code will have a high chance of being in single-cycle memory when the processor needs it
    • What comes next: The next set of instructions/data is on its way into the cache ready for next access
    • Benefit: Instruction cache almost always beneficial, data cache sometimes beneficial
    • Locality: If a word is just referenced, it is likely words in the vicinity will be referenced as well (one missed word results in several found words)
      • Example: Tool miss (need to go to the garage to get tools), then tool hits (gathered tool set avoiding future misses)

[edit] Concepts

  • Word: The number of bits a CPU can process at one time

[edit] How it works

  • Cache Read
    • Processor attempts to read word from memory
    • Cache check: A check is made to determine if the word is in the cache
      • Cache hit: Word delivered to the processor
      • Cache miss: Block of main memory read into cache, then just the requested word is delivered to the processor (more words next door)
  • Mapping
    • Fields: Break the main memory address into multiple fields
      • Location: Where data resides in the cache if it is resident/Where to be placed if it is not resident

[edit] Components

  • Ways and lines combine to form locations in cache where instructions/data are stored
    • Ways: Made up of lines
    • Lines: Collection of consecutive bytes (e.g. 32 bytes long)
  • Cache blocks: Contain multiple words
    • Valid bit: Specifies whether the block is valid (possible hit - may not be what you're looking for) or invalid (cache miss)


[edit] Mapping Types

  • Mapping: To enable the CPU to locate data copied into the cache (memory location -> cache location)
    • Provides: Storage location, method for locating stored data

[edit] Direct Mapped

Each memory location maps to a single cache line, shared with many other memory locations

  • Property: 1 Memory location only maps to 1 cache line - one end of the spectrum
  • Disadvantages: Very restrictive, only one (Only one of many addresses that share the line can use it at a given time), Unlikely in cache, Low performance gain, Thrashing
  • Advantages: Simple to implement, doesnt require any searching
  • Implementation: Map block X of main memory to block Y of cache mod N (N is the total number of blocks)
  • 1-Way Set Associative: Direct Mapped cache is a special cache of direct mapped cache with a set size of 1.
  • Memory address: [Tag] [Block] [Word]
    • Block: Signifies a unique block of the cache (how many bits to uniquely identify a block)
    • Word: Specifies a single word inside that block (how many bits needed to distinguish between words in a block)
    • Tag: Uniquely identifies that particular block (however many bits are left)
  • Cache line: [Tag] [Data]
  • How it works: Memory address split up into fields, use the block to locate the possible block, if its valid then possible cache hit. Check the cache line and see if the tag matches, if so its a cache hit.

[edit] Fully Assocative

Any memory location can be cached in any cache line

  • Property: 1 Memory location can map to any cache line - other end of the spectrum
  • Memory Address: [Tag] [Word]
    • Tag: Uniquely identifies a particular block (with a large tag)
    • Word: Specifies a single word inside that block
  • Cache line: [Tag] [Data] (Large tag stored with data)
  • Associative memory: Special hardware for associative searching (all in parallel)
  • How it works: Tag field of the memory address compared to all valid tag fields in cache. If a match is found it is a cache hit. Otherwise, use a replacement algorithm to replace a cache line.
  • Advantages: Always the best performance because very likely in cache, flexible
  • Disadvantages: Very expensive, requires searching, requires associative memory, requires cache replacement, less simple, large tag stored with data


[edit] N-Way Set Associative

The cache is split into sets of N lines each, a memory location can then be cached in any line inside a given set

  • Most common implementation. Typically 4-way associative for instructions
  • Properties: The size of N determines how many lines a memory address can map to. One memory location maps to N cache lines. Middle of the spectrum
  • Advantages: Very close to fully-associative performance
  • Disadvantages: Requires cache replacement, adds complexity
  • Memory Address: [Tag] [Set] [Word]
  • Cache entry: [Tag] [Data]
    • Tag: Uniquely identifies a particular block
    • Word: Specifies a single word inside that block
    • Set: Specifies which cache set through which to search
    • How it works: Set field of the memory address maps to a set of cache entries. Each cache entry is searched against the tag in parallel (associative search), and a cache hit occurs if there is a match.


[edit] Cache Replacement

Determening the victims to be replaced from the cache when new entries are needed

  • Direct replacement: Used with direct mapping - there can only be one mapping, and it is ejected
    • Evaluation: Not very flexible, often undesirable
  • Least recently used: System keeps history of accesses for each cache block, ejects the one with the least accesses
    • Evaluation: Requires maintaining list with lots of contents
  • FIFO: First in, first out. Eject the blocks that have been in the cache
    • Evaluation: Requires maintaining history, very recently used things ejected
  • Random: Select a victim at random and eject
    • Evaluation: Theoretically avoids thrashing

[edit] Cache Write Policies

  • Write-through: When update value in cache, write through to memory at the same time
    • Evaluation: Ensures cache and memory are always consistent. Each write requires access to main memory, slows down whole system, lots of memory access
  • Write-back: Only updates blocks in main memory when cache block needs to be evicted
    • Evaluation: Faster, less memory traffic. Main memory and cache may not contain the same value, cache data may be lost

[edit] More advanced

  • MESI
    • States: Modified/Exclusive/Shared/Invalid
      • Modified: Entry in this cache only is valid
      • Exclusive: No other cache has this cache line, memory is up to date
      • Shared: Other caches have the cacheline and the memory is up to date
      • Invalid: Cache does not contain valid data
    • How it works:
      • Reading if nobody else has it: Read it in and enter exclusive state
      • Reading if another has it and modified it: Other snoops request, tells the requester that it has changed, to wait and writes-back to memory. Requester requests it again. Both enter shared state.
      • Reading if another has it and hasnt modified: Read from memory, both into shared state
      • Modifying and shared: Modifies and changes to modified state, notifies others to change to Invalid state
      • Modfiying and not shared: Modifies and changes to modified state.
  • MOESI
    • Advantage: Caches with owner states update each others line with current data rather than writing back to main memory, saving bandwidth
    • States: Modified/Owner/Exclusive/Shared/Invalid
      • Shared: Either the main memory is up-to-date or there is somebody else in the owner state.
      • Owner: Entry in this cache is valid, memory is not up to date. Other caches may also have valid copies.
    • How it works
      • Same as MESI
      • Reading if another has it and modified it: CPU1 wants to read data modified by CPU2. CPU2 sends to CPU1 via point-to-point link, CPU2 enters Owner state and CPU1 enters shared state.
      • Modifying in shared state: Change it, notify all and enter Exclusive state.
      • Write-back: Write-back occurs when the owner would eject from cache

[edit] Problems

  • Thrashing: A cache line being constantly being replaced with another line, decreasing performance significanty
    • Bad locality
      • 2D arrays: Arrays are typically stored in row-major order.
        • Good: Request for the first row causes a miss, the block is transferred to the cache and subsequent accesses to that row are hits.
        • Bad: If one row fits in one cache block and the cache can hold all but one row of the array, accessing the array in a column-major order would produce misses each access.
        • Explanation: Data being transferred in for each row is not being used as the array is accessed by column and the cache is not large enough to hold the entire array.
    • Object oriented programming
    • Loop through linear array: Looping through a linear array

[edit] Where Caches go Bad

[edit] Breaks Direct

Arrays are likely to be aligned on high order bits, and thus would be competing for the same entries in the directly mapped cache.

float f[1024*1024*1024*2]; for (i = 0; i < 1024 * 1024 * 1024; i++) { f[i] = 2 * f[i + 1024 * 1024 * 1024]; }

[edit] Break 2-way Assoc

3 different items will be competing for the same 2 lines in a given set, thus driving one out. Every time through the loop, at least 1 gets ejected, which is bad. This leads to data cache collisions float f[1024*1024*1024*3]; for (i = 0; i < 1024 * 1024 * 1024; i++) { f[i] = 2 * f[i + 1024 * 1024 * 1024] + f[i + 1024 * 1024 * 1024 * 2]; }

How it breaks: 000101 = LOL 010101 = LAL 020101 = LIL TTSSWW


Set 00: -Free- -Free-

Set 01: [Tag 00] ACKLOLSYSPOPMOO [Tag 01] PCKLALSYIPIPMOI

At this point, the third entry would cause one of the previous 2 entries to be ejected.
