[edit] Pentium Pro

[edit] Dynamic Execution

  • Multiple branch prediction: Predicting which branches likely to be processed next to increase work fed to the processor
  • Dataflow analysis: Analyse which instructions dependent on other results/data to create optimised schedule
  • Speculative execution: Instructions carried out speculatively based on optimised schedule keeping superscalar processing busy

[edit] RISC translation

Translates from Intel instructions into RISC instructions which are executed. Emulates the Intel architecture (rather than direct execution)

[edit] Translation

  • Why: Wasn't possible to decouple sequentialisation imposed on assembly code by instruction set.

[edit] Components

  • Uops: Instructions are dynamically translated into Uops (Intel ISA (Instruction Set Architecture) to RISC uOps)
  • Out-of-order execution: Architectural instructions are translated, marked with availability of valid operands in registers, fed into out-of-order execution unit
  • Reservation station: Keep track of dependencies and ensure instructions don't fire until their dependencies are met
  • Fetch-decoder unit: Takes the instruction stream and translates it
  • Dispatch unit: Accepts the stream as a data flow stream - sequencing instructions based on availability of operands. Instructions fired based on graph of dependencies (not on sequence)
  • Retirement unit: Commits results to the permanent architectural state (some instructions do not commit at all, only work with dependencies), attempts to make things look like original order

[edit] Architecture

  • Seperate registers: Architectural registers, non-architectural registers
    • Physical but not published. Exist inside the out-of-order execution engine and are associated with reservation stations
  • Seperate caches: Seperate instruction and data caches for use by the Uops

[edit] How it works

  • Translation: Retrieve instructions and translate them into Uop instructions
  • Sit in pool: Instructions sit in a pool of RISC instructions waiting to be executed
  • Dispatch and execution: The instructions are then dispatched and executed when they are executable
  • Retirement: Reordering (to make sure things like interrupts work), putting it back together so it looks sequential
  • Speculation: Basic blocks aren't big enough to keep the units busy, so need to speculate ahead beyond branches, loops etc.

[edit] Branch Prediction

A branch predictor is a digital circuit that tries to guess which way a branch (e.g. an if-then-else structure) will go before this is known for sure. The purpose of the branch predictor is to improve the flow in the instruction pipeline.

  • Importance
    • Longer streams of instructions: Basic blocks aren't large enough so need to get a longer stream of contiguous instructions and guess past jumps
    • Unwind penalty: Bad penalty for wrong guesses - If guess wrong, have to unwind, flush the pipelines and restore state in non-architectural registers
      • Penalty: 11-15 cycles with around a 75 instruction uOp instruction penalty (40 instruction penalty in the underlying stream)

[edit] Components

  • Branch target buffer: Holds 512 entries with 4 bits of history
  • How it works if not in buffer: Naive branch prediction
  • If not in branch target buffer, do naive branch prediction
  • Predict taken for negative offset (loop)
  • Predict not-taken for positive offset (conditional)
  • Optimisation: Compiler can use this knowledge to put the most likely one first

[edit] Table entries

  • Tag: Address of the branch
  • Target: Contains where to branch to (can differ on different iterations)
  • History: 4 bit history (with 1 for taken, 0 for not taken). Used as an index to an associated 16 2-bit counter
    • Length: 4-bit history, but could have more to allow for more complex pattern detection
    • History of how the branch was last taken on the 4 last times
    • Updating: Updated each time you know what happened with the branch
  • 16 2-bit words: Saturated 2 bit counter
    • Represents 4 states: 00, 01, 10, 11
    • Counter: Each state has an associated saturated 2-bit counter
      • Increment (go up, up to 11) when the branch is taken
      • Decrement (go down, down to 00) when the branch is not taken
    • Usage
      • Predict not taken when counter is 00 or 01 (strong and weak not taken)
      • Predict taken when counter is 10 or 11 (weak and strong taken)
  • Justification: 2 bits so a single glitch doesnt spoil the prediction, but more bits don't add any significant benefit.
  • Analogy: Like table tennis serving. Hit once to serve, hit once to win the point. (Need to be wrong twice from strong prediction to predict differently)

[edit] How it works

  • 2 level system (history and counters)
  • Current history is used as an index to get an associated counter - you have a prediction counter for each different situation (each different history)
  • The counter specifies strong/weak predict taken/not taken
  • Follow the prediction of the counter and load them into the pipeline
  • When the result comes in, update the counter and the history

[edit] Evaluation

  • Can learn to recognise repetitive patterns (e.g. an alternating loop)
  • Can handle deviations from the normal pattern (from strong to weak)
  • Can handle mostly regular patterns with occasional deviations

[edit] Pentium 4

[edit] Properties

  • 20 stage pipeline
  • 4k entry branch target buffer, reduced misses by 33%
  • Double speed ALU, up to 4.4 GHz
  • 256/512k 8-way associative L2 cache on chip
  • 42M/55M transistors
  • 146/217mm^2 die
  • 126 instructions out-of-order window (up from 42)
  • 400Mhz 64-bit front size bus (up from 133Mhz)
  • L1 cache halved to 8k but latency reduced

[edit] Components

  • Execution trace cache: Instruction cache for decoded instructions (holds the uOp instructions)
    • Removes branches from RISC instructions and generates traces (sequences of instructions)
    • Loop unrolling, branch predictions etc on the fly
    • Speculation: Speculative code being generated and fed into the RISC engine
    • Allows branch prediction errors to recover at the micro-op level rather than at the x86 instruction level
    • Can hold up to 12k uOps
    • 1 IA instruction -> 1-4 uOps
    • Local referencing: References point to the cache not back to the architectural level instructions
  • Decoder: Generates uOps, typically around 10-30 deep

[edit] Pipelining

  • 3 pipelines
    • Instruction decoder pipeline: IA32 instruction level
    • uOp execution pipeline: Own branch predictor, works on trace cache addresses
    • Scheduler pipeline: Within each functional unit

[edit] Branch Prediction

  • Static prediction: The same as before - backwards taken, forward not
  • Properties
    • Much larger Branch Target Buffer (4mb over 512)
    • Static branch hint prefixes: Compiler hinting: Compiler can give branch prediction hints attached to conditionals (2EH - branch not taken, 3EH branch taken)
    • Higher penalty on mis-prediction
    • Average mis-prediction rate of ~6%
    • Register renaming: Using different registers to avoid output dependencies, putting back into correct registers at the end
    • Less need for re-order buffer (in earlier architectures, holds words pushed out of RISC processor into re-order buffer) - bigger register file avoids this

[edit] Execution engine

  • Load/store ports: Memory load and memory store
  • Several execution ports: Waiting in buffer on port, execute as operands become available
    • ALUs operate at twice the speed
    • Execution ports have different execution units (e.g. ALU, FP, Int operations, memory load/store etc.)
    • Different operations go down different execution ports
    • The number of execution ports controls the number of operations you can execute at once
  • Guessing: If wait until registers have actual values, slows down pipeline. Instead, guess when required values present and send in based on those guesses.

[edit] Problems

  • Round and round: Instructions not ready don't stall the pipeline, but ripple through the pipeline not doing anything (they are anulled) until the end of the pipe. Then they get put back into upstream pipeline and executed again
    • Lots of instructions going round and round without doing anything
    • Multiple replays: Once commit to a RISC instruction, a single instruction can replay lots and lots of times
    • Heat: It gets hotter and hotter
    • Sends without guarantee of execution: Instructions are sent for execution with no guarantee that they can be executed
